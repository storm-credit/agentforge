# 에이전트 빌드 정의

## 정의

에이전트 빌드는 특정 업무 목적을 수행하도록 모델, 지식, 도구, 권한, 승인 정책, 실행 흐름, 평가 기준, 로그 정책을 하나로 묶어 배포 가능한 상태로 만든 실행 계약이다. Agent Builder가 만드는 결과물은 단순 프롬프트나 코드 조각이 아니라 Runtime Orchestrator가 검증하고 실행할 수 있는 버전 고정 패키지여야 한다.

MVP에서 에이전트 빌드는 다음 성격을 가진다.

- **계약 단위**: 입력, 출력, 권한, 지식 범위, 도구 호출, 로그 필드를 스키마로 고정한다.
- **배포 단위**: 승인된 `build_id`만 런타임에 활성화한다.
- **감사 단위**: 어떤 모델, 문서 인덱스, 프롬프트, 정책으로 답했는지 재현 가능한 메타데이터를 남긴다.
- **품질 게이트 단위**: 빌드 전/후 검증을 통과하지 못하면 배포하지 않는다.

## 식별자와 버전 규칙

| 필드 | 의미 | 규칙 |
|---|---|---|
| `schema_version` | Agent Card 스키마 버전 | MVP는 `agentforge.agent_card/v1` |
| `agent_id` | 논리 에이전트 ID | 영문 소문자, 숫자, `_`, `-`만 허용 |
| `version` | 사람이 관리하는 릴리스 버전 | SemVer 권장 |
| `build_id` | 불변 빌드 ID | `agent_id:version:content_hash` 형태 권장 |
| `index_snapshot_id` | 참조하는 문서 인덱스 스냅샷 | RAG 재현성을 위해 필수 |
| `status` | 빌드 상태 | `draft`, `review`, `approved`, `deployed`, `retired` |

`agent_id`가 같아도 `version`, `build_id`, `index_snapshot_id`가 다르면 다른 실행 계약으로 취급한다. 운영 중인 빌드는 직접 수정하지 않고 새 빌드를 생성한다.

## 계약 구성 요소

```text
Agent Build
├─ Identity Contract: 에이전트 목적, 소유 부서, 대상 사용자
├─ Input/Output Contract: 허용 입력, 거절 입력, 응답 형식
├─ Model Contract: 모델 후보, 파라미터, fallback, 토큰 제한
├─ Prompt Contract: system/developer prompt, 금지 지시, 변수
├─ Knowledge Contract: 참조 컬렉션, 인덱스 스냅샷, freshness 정책
├─ Retrieval Contract: top-k, ACL filter, rerank, citation 정책
├─ Tool Contract: 허용 도구, 파라미터 스키마, timeout, 승인 필요 여부
├─ Security Contract: prompt injection, PII, 기밀 등급, 외부 반출 제한
├─ Runtime Flow Contract: Planner/Retriever/Critic/Security Guard 단계
├─ Observability Contract: run log, audit log, metric, 보존 기간
└─ Quality Gate Contract: 빌드 검증, 런타임 차단, 회귀 평가
```

## Runtime 요청/응답 계약

### AgentRunRequest

```yaml
request_id: "run_20260509_000001"
agent_id: "security_policy_assistant"
build_id: "security_policy_assistant:1.3.0:sha256-8b9c..."
user:
  user_id: "u12345"
  department_id: "security"
  roles: ["employee", "policy_reader"]
  clearance_level: "internal"
  project_scopes: ["corp-policy"]
input:
  message: "외부 협력사에 로그 파일을 전달해도 되나요?"
  locale: "ko-KR"
  channel: "agent_studio"
runtime_options:
  dry_run: false
  max_latency_ms: 8000
  trace_level: "standard"
```

필수 검증:

- `build_id`가 배포 상태인지 확인한다.
- 사용자 권한 컨텍스트를 사내 IAM/SSO에서 재조회한다. 클라이언트가 보낸 권한을 신뢰하지 않는다.
- `message` 원문은 감사 저장소에 평문 저장하지 않고 정책에 따라 해시 또는 암호화 저장한다.

### RuntimeState

```yaml
run_id: "run_20260509_000001"
build_id: "security_policy_assistant:1.3.0:sha256-8b9c..."
stage: "retrieval"
intent:
  category: "policy_qa"
  risk_level: "medium"
plan:
  - step_id: "retrieve_policy"
    agent: "Retriever"
    required: true
retrieval:
  acl_filter:
    departments: ["security", "all"]
    clearance_lte: "internal"
  candidates: []
draft_answer: null
critic_findings: []
security_flags: []
citations: []
```

RuntimeState는 내부 상태이며 사용자에게 직접 노출하지 않는다. 각 단계는 이 상태에 append-only 방식으로 결과를 추가하고, 이전 단계 결과를 덮어쓸 때는 사유를 로그에 남긴다.

```yaml
run_id: "run_20260509_000001"

guardrails:
  blocked: false
  flags: []

  groundedness: "pass"
  citation_coverage: "pass"
  acl_checked: true
```

## Runtime Flow 계약

MVP의 기본 흐름은 다음 순서를 따른다.

```text
Ingress
-> Security Guard(pre-check)
-> Planner
-> Retriever
-> Answer Generator
-> Critic
-> Security Guard(final-check)
-> Response Formatter
-> Audit Logger
```

### 1. Security Guard pre-check

- 입력의 prompt injection, 비인가 데이터 요청, 개인정보/기밀 반출 의도를 먼저 점검한다.
- 차단 조건이면 Retriever나 모델 생성 단계로 넘기지 않는다.
- 애매한 경우 `risk_level=medium`으로 표시하고 Planner가 안전한 답변 경로를 선택하게 한다.

### 2. Planner

- 사용자 요청을 `policy_qa`, `document_summary`, `comparison`, `procedure_help`, `unsupported` 같은 업무 intent로 분류한다.
- 필요한 지식 소스와 검색 전략을 결정한다.
- 허용되지 않은 도구나 문서 범위를 요구하는 계획은 만들 수 없다.
- 결과는 실행 가능한 step list여야 하며 자연어 추론 메모를 그대로 저장하지 않는다.

### 3. Retriever

- Planner가 지정한 knowledge source에 대해 사용자 권한 컨텍스트를 적용한 검색을 수행한다.
- ACL 필터는 vector search 이전 payload filter로 적용한다.
- 검색 결과에는 `chunk_id`, `document_id`, `score`, `acl_decision`, `citation_locator`를 포함한다.
- 근거가 부족하면 빈 컨텍스트를 반환하고 Generator가 추측하지 않게 한다.

### 4. Answer Generator

- 허용된 chunk만 컨텍스트로 받아 답변 초안을 만든다.
- 답변 문장 중 정책/절차/수치/날짜/예외 조건에는 citation을 붙인다.
- 근거가 없거나 권한상 볼 수 없는 내용은 "확인 가능한 근거가 없습니다"로 응답한다.

### 5. Critic

- 답변이 검색 근거에 실제로 의해 뒷받침되는지 확인한다.
- citation 누락, 과잉 일반화, 오래된 문서 사용, 충돌하는 근거를 잡는다.
- 수정 가능하면 Generator에 재작성 요청을 보내고, 반복 한도를 넘으면 안전한 실패 응답으로 전환한다.

### 6. Security Guard final-check

- 최종 답변에 기밀 원문 과다 노출, 개인정보, 권한 밖 문서명, prompt injection 반영이 없는지 검사한다.
- 위험 플래그가 있으면 마스킹, 축약, 차단 중 하나로 처리한다.
- 모든 차단/마스킹 결정은 audit log에 남긴다.

schema_version: "agentforge.agent_card/v1"

## MVP Quality Gates

### 빌드 시점 게이트

- Agent Card가 JSON Schema/YAML Schema를 통과해야 한다.
- `knowledge_sources[].index_snapshot_id`가 존재하고 비활성/삭제 상태가 아니어야 한다.
- ACL 필수 payload 필드가 인덱스 샘플 chunk 100%에 있어야 한다.
- prompt ref, model ref, tool ref가 사내 registry에서 resolve되어야 한다.
- golden set 질의에서 citation coverage, ACL 차단, injection 방어 케이스가 통과해야 한다.
- 배포 승인자는 에이전트 소유 부서와 보안 검토자를 모두 포함해야 한다.

### 런타임 게이트

- Security Guard pre-check가 차단한 요청은 Planner 이후 단계로 진행하지 않는다.
- Retriever는 `acl_filter_applied=true` 없이는 결과를 반환할 수 없다.
- 근거가 필요한 문장에 citation이 없으면 Critic이 재작성 또는 실패 처리한다.
- 최종 답변은 Security Guard final-check를 통과해야 한다.
- run log와 audit log 기록 실패 시 응답은 성공 처리하지 않는다.

## 빌더가 만드는 것

Agent Builder는 에이전트 자체의 코드만 만드는 것이 아니라, 실행 가능한 설정과 정책 묶음을 만든다. MVP에서는 코드 생성보다 **검증된 Agent Card 생성**, **문서 인덱스 연결**, **권한/근거 기반 런타임 계약 생성**이 핵심이다.
