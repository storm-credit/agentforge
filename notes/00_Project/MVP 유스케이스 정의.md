## 1. 유스케이스 요약

관리자가 승인된 사내 문서 묶음과 접근 정책을 선택해 RAG 기반 Q&A 에이전트를 만들고, 사용자는 본인 권한으로 접근 가능한 문서 범위 안에서 근거가 포함된 답변을 받는다.

## 2. 왜 이 유스케이스가 MVP인가

| 판단 기준 | 이유 |
|---|---|
| 업무 수요 | 규정, 매뉴얼, 보안 지침, 운영 절차는 반복 문의가 많다 |
| 보안 검증 | 문서 ACL, 부서 권한, 문서등급 필터를 실제로 검증할 수 있다 |
| 기술 검증 | 수집, 파싱, 임베딩, 검색, rerank, citation, 로그의 핵심 흐름이 모두 포함된다 |
| 확장성 | 이후 DB 조회, 그룹웨어, ERP 연동 에이전트의 기본 실행/감사 구조로 재사용된다 |
| 일정 현실성 | 쓰기 작업과 복잡한 외부 시스템 연동 없이 8주 내 데모가 가능하다 |

## 3. 주요 페르소나

| 페르소나 | 역할 | 성공 기준 |
|---|---|---|
| Agent 관리자 | 문서 묶음과 정책을 설정해 에이전트를 생성한다 | 반나절 이내 파일럿 에이전트 설정 |
| 문서 소유자 | 등록 가능한 문서와 접근 등급을 확인한다 | 오래된 문서와 민감 문서를 구분 |
| 일반 사용자 | 업무 질문을 입력하고 근거가 있는 답변을 받는다 | 문서 검색 시간을 줄이고 답변 출처 확인 |
| 보안/감사 담당자 | 권한, 로그, 민감정보 노출 여부를 검토한다 | 권한 없는 문서 노출 0건 |
| 플랫폼 운영자 | 색인, 모델, 실행 로그, 장애를 확인한다 | 장애 원인과 재처리 대상 식별 |

## 4. MVP 대상 업무 후보

| 후보 | 설명 | MVP 적합도 | 비고 |
|---|---|---|---|
| 보안/개인정보 규정 Q&A | 임직원이 보안 규정, 반출 기준, 개인정보 처리 기준을 질의 | 높음 | 권한/감사 검증에 적합 |
| 인사/총무 업무 매뉴얼 Q&A | 휴가, 출장, 복리후생, 비용 처리 기준 질의 | 높음 | 반복 문의 감소 효과가 명확 |
| IT 운영 절차 Q&A | 장애 대응, 계정 신청, 사내 시스템 이용 절차 질의 | 중간 | 문서 최신성 관리가 중요 |

MVP 1차 데모는 위 후보 중 1개를 선택해 끝까지 연결한다. 현재 권장 후보는 "보안/개인정보 규정 Q&A" 또는 "인사/총무 업무 매뉴얼 Q&A"이다. 최종 선택은 파일럿 부서와 문서 확보 가능성에 따라 1주차에 확정한다.

## 5. 사용자 여정

| 단계 | 행위자 | 행위 | 시스템 동작 | 산출/로그 |
|---|---|---|---|---|
| 1 | 문서 소유자 | 파일럿 문서 후보와 등급을 제공 | 문서 메타데이터 등록 준비 | 문서 목록 |
| 2 | 관리자 | 문서 묶음을 등록 | 파싱, 청킹, 임베딩, 색인 수행 | 색인 상태 로그 |
| 3 | 관리자 | 에이전트 목적, 모델, 답변 정책, 접근 부서를 설정 | Agent Card와 정책 버전 저장 | 에이전트 v1 |
| 4 | 보안 담당자 | 권한 테스트 계정으로 접근 검증 | ACL 필터와 차단 로그 확인 | 보안 점검 결과 |
| 5 | 사용자 | 업무 질문 입력 | 권한 확인 후 허용 문서만 검색 | 질문/검색 로그 |
| 6 | 시스템 | 답변 생성 | 근거 citation과 함께 응답 | 답변/근거 로그 |
| 7 | 사용자 | 도움이 되었는지 피드백 | 평가 데이터로 저장 | 피드백 로그 |
| 8 | 운영자 | 실패/저품질 케이스 확인 | 재색인, 정책 수정, 문서 보강 | 개선 백로그 |

## 6. 기능 요구사항

### 6.1 Agent Builder

- 에이전트 이름, 설명, 소유 부서, 담당자를 입력할 수 있어야 한다.
- 문서 묶음, 모델, 시스템 프롬프트, 답변 정책을 선택할 수 있어야 한다.
- 공개 범위를 부서, 사용자 그룹, 문서등급 기준으로 설정할 수 있어야 한다.
- 에이전트 상태를 draft, review, published, archived로 관리할 수 있어야 한다.
- 변경 시 버전과 변경 사유가 남아야 한다.

### 6.2 문서 수집과 색인

- PDF, DOCX, XLSX, TXT, Markdown 파일을 기본 대상으로 한다.
- 문서명, 소유 부서, 문서 버전, 등급, 접근 그룹, 등록일을 메타데이터로 저장한다.
- 색인 실패 문서와 실패 사유를 관리자가 확인할 수 있어야 한다.
- 문서가 갱신되면 재색인 대상임을 표시해야 한다.
- MVP에서는 대용량 전사 문서 동기화보다 파일럿 문서 묶음의 정확한 색인을 우선한다.

### 6.3 검색과 답변

- 사용자의 부서/그룹/문서등급 권한을 기준으로 검색 후보를 먼저 제한한다.
- 답변에는 문서명, 섹션, 페이지 또는 원문 위치가 citation으로 표시되어야 한다.
- 근거가 부족하면 추측하지 않고 추가 확인이 필요하다고 답해야 한다.
- 서로 다른 문서의 기준이 충돌하면 충돌 가능성을 알리고 근거를 나란히 제시한다.
- 동일 질문에 대한 검색 결과, 답변, 모델, 정책 버전이 추적 가능해야 한다.

### 6.4 운영과 감사

- 관리자 화면에서 문서 색인 상태, 에이전트 상태, 최근 실행 로그를 볼 수 있어야 한다.
- audit log에는 사용자, 시각, 질문, 검색 문서, 답변 요약, 모델, 정책 버전, 오류 여부가 남아야 한다.
- 로그에 민감정보 원문을 과도하게 저장하지 않도록 저장 항목과 마스킹 기준을 분리한다.
- 파일럿 기간에는 실패 질문과 낮은 평가를 수집해 개선 백로그로 전환한다.

## 7. 비기능 요구사항

| 영역 | 요구사항 |
|---|---|
| 보안 | 권한 없는 문서는 검색 후보에 포함되지 않아야 한다 |
| 신뢰성 | 색인 실패, 모델 오류, 검색 결과 없음 상태가 사용자/관리자에게 구분되어야 한다 |
| 성능 | 파일럿 문서 묶음 기준 일반 질의 응답은 업무 사용이 가능한 수준으로 제공한다 |
| 감사 | 주요 실행 단계와 정책 버전을 재현 가능한 수준으로 남긴다 |
| 운영성 | 운영자가 실패 원인, 재색인 대상, 저품질 질문을 확인할 수 있어야 한다 |
| 사용성 | 관리자는 개발자 도움 없이 기본 에이전트를 생성할 수 있어야 한다 |

## 9. 수용 기준

| ID | 기준 | 검증 방법 |
|---|---|---|
| AC-001 | 관리자가 문서 묶음과 답변 정책을 설정해 에이전트를 생성할 수 있다 | 관리자 시나리오 테스트 |
| AC-002 | 사용자는 본인 권한 문서만 검색 결과로 받을 수 있다 | 권한별 테스트 계정으로 질의 |
| AC-003 | 권한 없는 문서는 citation, 답변, 로그 조회 화면에 노출되지 않는다 | 보안 테스트 |
| AC-004 | 답변에는 문서명과 위치 정보가 포함된 citation이 표시된다 | 평가 질문 세트 |
| AC-005 | 근거가 부족한 질문에는 추측 답변 대신 한계 안내를 제공한다 | 부정 질문 테스트 |
| AC-006 | 실행 로그에서 질문, 검색 문서, 답변 상태, 정책 버전을 확인할 수 있다 | 운영자 점검 |
| AC-007 | 색인 실패 문서와 오류 사유를 관리자가 확인할 수 있다 | 실패 파일 업로드 테스트 |
| AC-008 | 파일럿 평가 질문 기준 유용 답변 비율이 80% 이상이다 | QA/Eval 결과 |

## 10. 평가 데이터 구성

- 파일럿 문서 30~100개 내외를 1차 대상으로 한다.
- 평가 질문은 사실 확인형, 절차 안내형, 조건 비교형, 권한 차단형, 근거 부족형으로 나눈다.
- 각 질문은 기대 답변, 허용 문서, 금지 문서, citation 기준을 함께 정의한다.
- 평가 결과는 정답성, citation 적합성, 권한 준수, 답변 명확성으로 기록한다.

## 11. 제외 시나리오

- 사용자가 에이전트에게 결재를 올리거나 메일을 보내게 하는 작업
- ERP/그룹웨어 데이터를 쓰기 변경하는 작업
- 관리자 승인 없이 전체 사내 문서를 자동 수집하는 작업
- 사용자가 볼 수 없는 문서의 존재 여부를 암시하는 답변
- 근거 없는 일반 지식 답변을 사내 기준처럼 제시하는 답변
