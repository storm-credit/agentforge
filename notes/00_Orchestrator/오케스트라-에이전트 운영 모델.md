# 오케스트라-에이전트 운영 모델

## 짧은 답

맞다. Agent Forge의 작업 방식은 `오케스트라가 총괄하고, 전문 에이전트가 진행한다`는 구조다.

오케스트라는 방향, 순서, 게이트, 통합, 충돌 조정을 맡는다. 각 전문 에이전트는 PM, 아키텍처, 보안, RAG, 백엔드, 프론트엔드, DevOps, QA 같은 자기 영역을 깊게 맡는다.

## 에이전트는 딥한가

목표는 딥하다. 단순히 이름만 붙은 역할이 아니라, 각 에이전트가 자기 분야의 판단 기준, 체크리스트, 리스크, 산출물 품질 기준을 가져야 한다.

예를 들면:

- 보안 에이전트는 권한 누락, 감사 누락, 프롬프트 인젝션, 문서 노출 가능성을 계속 의심한다.
- RAG 에이전트는 chunking, ACL filtering, citation, evaluation corpus를 함께 본다.
- 백엔드 에이전트는 API 계약, DB migration, job 상태, audit 저장을 연결한다.
- QA/Eval 에이전트는 좋은 말이 아니라 측정 가능한 테스트와 release gate로 바꾼다.

## 깊이 단계

| 단계 | 이름 | 의미 |
|---|---|---|
| D0 | 이름만 있는 에이전트 | 역할명은 있지만 결과가 일반적이다 |
| D1 | 구조화된 초안 에이전트 | 자기 역할의 체크리스트나 초안을 만든다 |
| D2 | 딥 전문 에이전트 | 도메인 규칙, 리스크, 대안, 완료 기준까지 제시한다 |
| D3 | 검증 전문 에이전트 | 구현/문서가 테스트, 보안, release gate를 통과하는지 검증한다 |
| D4 | 운영 전문 에이전트 | 결정 기억, 회귀 감시, 운영 지표 갱신까지 맡는다 |

현재 목표는 설계 작업에서는 D2, 구현/릴리스 작업에서는 D3다. D4는 추후 telemetry, 평가 이력, decision memory가 제품 안에 연결된 뒤의 목표다.

## 현재 적용 방식

현재 저장소에서는 다음 방식으로 운영 모델을 구현한다.

- `notes/00_Orchestrator`: 총괄, 디스패치, 의사결정
- `notes/01_PM`부터 `notes/09_QA_Eval`: 전문 에이전트별 작업장
- `docs/`: 합의된 공식 산출물
- `apps/`: Sprint 0부터 실제 실행 가능한 구현 골격
- `deploy/compose`: 로컬 실행과 폐쇄망 배포로 이어질 운영 골격

## 오케스트라의 역할

- MVP 범위를 고정한다.
- 어떤 에이전트가 어떤 산출물을 만들지 배정한다.
- 전문가 산출물 사이의 충돌을 정리한다.
- 보안/권한/감사 기준을 통과하지 못한 작업을 막는다.
- 초안을 공식 문서, backlog, code, runbook으로 승격한다.

## 전문 에이전트의 역할

- 자기 영역의 산출물을 깊게 만든다.
- 가정과 리스크를 명시한다.
- 다음 전문가에게 넘길 입력을 만든다.
- 완료 기준을 테스트 가능하게 쓴다.
- 오케스트라가 판단할 수 있도록 선택지와 근거를 제시한다.

## 제품으로 확장될 때

나중에 Agent Forge 제품 안에서는 이 모델이 다음 기능으로 바뀐다.

- Agent template
- domain checklist
- approval gate
- evaluation report
- audit-backed runtime trace
- tool risk rating
- release workflow

즉, 지금은 프로젝트 운영 방식으로 먼저 쓰고, 이후에는 Agent Forge 자체가 제공하는 제품 기능으로 흡수한다.
