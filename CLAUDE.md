# CLAUDE.md — AgentForge 작업 지침

사내망용 **통제형 AI 에이전트 빌더 플랫폼**(문서 기반 RAG). 개요·실행은 [ONBOARDING.md](ONBOARDING.md), 설계/계획은 `docs/superpowers/specs|plans/`.

## 작업 운영 규칙 (반드시)

1. **한 슬라이스 단위로 진행**: brainstorming → writing-plans → subagent-driven-development(TDD) → finishing-a-development-branch(push + PR + merge). 슬라이스는 작게, 백엔드/프론트 무관 영향 최소.
2. **매 작업(슬라이스)이 끝날 때마다, 턴 마지막에 "다음 작업 핸드오프 프롬프트"를 작성한다.** 새 세션에 그대로 붙여 실행 가능한 자기완결형이어야 한다.

   **2-a. 백로그는 전문가 패널로 도출한다(단독 판단 금지).** 슬라이스 종료 시 `dispatching-parallel-agents`로 직무별 에이전트(보안 WS2/RAG WS3/백엔드 WS4/프론트 WS7/DevOps WS8/QA·PM WS5·WS1 — 각 `notes/0X_*` 작업지시서 매핑)를 병렬로 띄워, 각자 "지금 코드로 닫을 수 있는 최고가치 항목"을 내게 하고 종합한다. 입력 기준은 항상 `notes/01_PM/WBS.md`(워크스트림/게이트) + 실제 상태 단일 출처 [docs/status-and-go-no-go.md](docs/status-and-go-no-go.md). 임의 리스트 금지. 범위가 좁으면 관련 직무만 부분 패널로 돌려도 되나, 종합 출처를 핸드오프에 명시. (패널 결과로 status 문서의 🔧/⛔ 분류를 갱신한다.)

   **2-b. 핸드오프 프롬프트 필수 섹션(이 순서·이 항목들):**
   - **제목** — `다음 작업: <무엇> (WS태그)`
   - **종합 판단 출처** — 전문가 패널(WS별 + notes/0X_* 작업지시서) 병렬 종합임을 명시(단독 아님)
   - **남은 작업 체크** — WBS+status 도출. 각 항목에 WS태그. 🔧 코드-now(진행률 N/M) / 🔧 측정·무결성 / 🔧 배포 / ⛔ 조직·모델·인프라 대기(SSO·실문서·폐쇄망·사내모델) 구분
   - **배경** — 직전 머지 PR 요약·현 main·baseline 테스트 수·핵심 파일/메모리(`[[...]]`) 링크
   - **목표/범위(YAGNI)** — 이번 슬라이스 한정 + 범위 밖 명시
   - **완료 기준** — pytest 풀스위트 그린(.env 옆으로, baseline 명시)·ruff·필요시 security-review/eval before-after·라이브 1회·status 문서 갱신·정직 단서
   - **활용 도구** — superpowers 체인, agentmemory recall/remember, Explore/subagent, security-review·requesting-code-review, WebSearch, eval 하네스
   - **환경/겟차** — 아래 "환경/겟차" 전부
   - **작업 끝나면** — memory:remember + status 갱신 + **전문가 패널 재실행** + 다음 핸드오프 + 완결표현 금지([[completion-claims-discipline]])

   **2-c. 완결 조건**: 패널이 코드-now 항목을 더는 못 내면 다음 프롬프트를 만들지 말고 **"코드 완결"을 선언**(남은 건 비코드 의존이라 명시). 코드-now 항목이 하나라도 있으면 "제품 완성/완결" 표현 금지. 무한히 새 작업을 만들어내지 않는다.

   **2-d. 백로그 실행은 `Workflow` 오케스트레이션으로, 원칙은 순차(사용자 확정 2026-07-08 — 이후 세션도 계속 이 방식 적용).**
   - **대전제(사용자 확정 2026-07-14, 요청 표현과 무관하게 고정)**: 사용자가 요청을 어떻게 적든(짧게/모호하게/기능 한 줄로 적어도) **모든 실작업은 오케스트레이터(현재 세션)가 총괄하고, 구현은 직무별 전문가 서브에이전트에게 "딥하게" 배정한다** — 각 에이전트에 역할·맥락·대상 파일/라인·완료 기준·환경 겟차·정직 단서까지 상세 브리핑하고, **작업 성격에 맞는 모델을 명시 지정**해 진행. 오케스트레이터가 인라인 단독으로 처리하지 않는다(전문가 패널/서브에이전트로 분해 → PR 오픈 → 오케스트레이터가 리뷰+머지). **예외(비례성)**: git 상태 확인·문서 한 줄 수정·단발 조회처럼 서브에이전트가 과한 사소 작업은 인라인 처리 가능하되, 애매하면 서브에이전트 경로를 택한다.
   - **모델 배정**: 기본 모델은 **Fable**(총괄/일반 작업). 보안·인가·정확성이 중요한 슬라이스(RBAC, ACL, 인젝션 가드 등)는 **opus**로 배정. 그 외 슬라이스는 작업 성격에 맞게 상황별 배정(예: 순수 인프라/문서=sonnet도 가능) — "나머지는 알맞게" 원칙.
   - **순차 vs 병렬**: 기본은 **한 슬라이스씩 순차** 실행(Workflow 단일 트랙) — 하나 완료(리뷰+머지)까지 확인 후 다음 슬라이스 착수. **여러 트랙 병렬 배치는 CI 워크플로가 갖춰지기 전엔 지양**(2026-07-08 배치에서 병렬 서브에이전트들이 공유 venv를 사고로 삭제한 전례 — 회귀 안전망 없이 여러 트랙을 동시에 돌리면 이런 사고를 자동으로 못 잡음). CI 도입 후엔 독립적이고 blast-radius 작은 도메인 슬라이스에 한해 병렬 배치 재개 가능.
   - **에이전트에게 절대 자동 머지를 맡기지 않는다**: 각 슬라이스 에이전트는 PR 오픈까지만 하고 멈추게 지시. 오케스트레이터(현재 세션)가 diff 검토(보안 슬라이스는 별도 security-review 에이전트) 후 직접 머지.
   - **공유 리소스 주의**: 격리 워크트리 에이전트가 venv/node_modules 접근이 필요하면 "메인 체크아웃 실행파일을 절대경로로 직접 호출"을 우선 지시하고, 디렉토리 정션 트릭은 꼭 필요할 때만 + "정션 자체만 제거, 재귀삭제 금지"를 명시.
   - **문서 전용 커밋도 브랜치+PR 경유**: main 직접 푸시는 자동 승인 시스템이 비일관적으로 차단할 수 있음 — docs-only 변경도 항상 브랜치+PR로.
3. **정직 우선**: 테스트·검증 결과를 있는 그대로. 측정 안 한 건 "안 했다"고. 약점/한계 명시.
4. **품질을 측정으로 입증**: 검색·답변 품질 변경은 `eval/harness/run_live_eval.py`로 before/after 수치. 보안 변경은 라이브 재현 + 가능하면 적대적 스윕.

## 제품 탐색·설계·구현 전 검증 (필수)

5. **작업 착수 전 먼저 질문한다.** 저장소·이슈·현재 대화에서 이미 확인된 사실은 다시 묻지 않는다. 구현 결과를 바꿀 정도의 미확정 사항이 있으면 가장 영향이 큰 질문부터 짧게 묻고, 답을 받기 전에는 해당 결정을 코드로 고정하지 않는다. 사용자가 판단을 위임한 경우에는 오케스트레이터가 합리적 기본안을 선택하되 `확정 사실 / 가정 / 미확정`을 구분해 기록한다. 중대한 미확정 사항이 없으면 `추가 질문 불필요`와 그 근거를 남기고 진행한다.

   **5-a. 사용자 인터뷰:** 신규 제품 흐름, 주요 UI, 권한·데이터 정책, 평가 기준을 정하는 슬라이스는 구현 전에 사용자를 인터뷰한다. 한 라운드당 3~7개의 구체적 질문으로 현재 업무 방식, 불편, 우회 방법, 핵심 사용자, 성공 기준, 실패로 보는 상황, 반드시 제외할 범위를 확인한다. 답변은 Issue 또는 설계 문서에 `User Interview Notes`로 요약하고, 기존 답변을 반복해서 묻지 않는다.

   **5-b. 맹점 훑기:** 계획 확정 전과 PR 머지 전 각각 한 번 수행한다. PM·보안·RAG/Data·Backend·Frontend/UX·Platform·QA/Eval 관점에서 다음을 확인한다: 권한 우회, 데이터 유출·보존·삭제, 실패·재시도·롤백, 동시성·멱등성, 마이그레이션·호환성, 관측성·감사, 성능·비용, 접근성·오류 UX, 테스트 사각지대, 운영자 오용, 범위 팽창. 발견 항목은 `FIX-NOW / QUESTION / MONITOR / OUT-OF-SCOPE`로 분류한다. 보안·데이터 무결성·릴리스 차단급 맹점이 미해결이면 HOLD한다.

   **5-c. 디자인 시안 4개를 한눈에 비교:** 신규 또는 대폭 변경되는 사용자 화면은 본 구현 전에 서로 실질적으로 다른 4개 시안을 하나의 비교 화면 또는 문서에 나란히 제시한다. 단순 색상 변형 4개는 인정하지 않는다. 각 안은 정보 구조, 탐색 방식, 화면 밀도, 핵심 작업 동선, 오류/빈 상태, 접근성, 구현 비용의 차이를 보여야 한다. 전체 구현 전에 사용자의 선택을 받으며, 사용자가 선택을 위임하면 PM이 한 안을 고르고 근거와 포기한 대안을 기록한다. Backend-only·내부 리팩터링은 `N/A` 근거를 남긴다.

   **5-d. 본보기 코드 조사:** 새로운 아키텍처·RAG·권한·Agent Builder·UI 패턴을 도입하기 전 GitHub에서 유사한 오픈소스 구현을 최소 3개 조사한다. 각 참고 대상에 저장소 URL, 라이선스, 유지보수 상태, 확인한 정확한 파일/경로와 가능하면 commit SHA, 채택할 패턴, 채택하지 않을 부분과 이유를 기록한다. 라이선스가 불명확하거나 호환되지 않는 코드는 복사하지 않는다. 참고 코드는 설계 입력일 뿐이며 이 저장소의 제품·보안·ADR 기준보다 우선하지 않는다. 코드 전체를 베끼지 말고 필요한 패턴을 독립적으로 구현한다.

   **5-e. 계획 이탈·막힘 기록:** 원래 계획과 달라지거나 막히면 숨기거나 임의로 우회하지 않는다. Issue/PR의 `Plan Deviations` 섹션에 다음을 기록한다: 원래 계획, 실제 발견 위치(파일·모듈·단계), 무엇이 달랐는지, 원인, 영향 범위, 검토한 선택지, 선택한 결정과 이유, 추가·변경된 테스트, 롤백/비활성화 방법, 남은 위험과 사용자 결정 필요 여부. 범위·보안 경계·데이터 모델·외부 의존성·완료 기준이 바뀌면 Work Order/ADR을 갱신하고 승인 또는 명시적 위임 없이 계속 구현하지 않는다. 사소한 구현 상세 변경은 진행할 수 있지만 PR에 반드시 남긴다.

   **5-f. 기본 진행 순서:** 기존 맥락 확인 → 필요한 선행 질문/사용자 인터뷰 → 맹점 훑기 → UI라면 디자인 4안 비교 → 오픈소스 본보기 조사 → Work Order/계획 확정 → TDD 구현 → 테스트·평가 → 계획 이탈 기록 확인 → 전문가 리뷰 → PR. 각 단계가 적용되지 않으면 생략하지 말고 `N/A` 사유를 기록한다.

## 환경/겟차 (로컬, Windows)

- **Python은 반드시 `apps/api/.venv/Scripts/python.exe`** (전역 python은 contract 테스트 silent skip).
- **uvicorn에 `--reload` 없음** → 백엔드 코드 변경 후 반드시 재기동. 라이브가 구버전 응답하면 의심. 포트 8000을 옛 프로세스가 점유하면 새 API가 exit 1 → 기존 python 정리 후 기동.
- **pytest 풀스위트는 `apps/api/.env`를 잠시 옆으로 옮긴 뒤 실행**(실 LLM/DB가 2개 테스트에 새어 실패) → 끝나면 복원. 현재 기준 141 passed / 0 skipped(CI에서도 동일 확인, PR #53).
- **라이브 스택**: `docker start agentforge-ollama compose-postgres-1 compose-qdrant-1` (볼륨 보존 — 모델 bge-m3/qwen3:1.7b, DB `agentforge_mvp2`, Qdrant `chunks_active` 유지). `.env` = qdrant 백엔드 + bge-m3 + agentforge_mvp2 + `AGENT_FORGE_RETRIEVAL_MIN_SCORE=0.53`.
- **프론트 npx/.bin 셰임 깨짐** → node 직접: dev `node node_modules/next/dist/bin/next dev <apps/web 절대경로> -p 3300`, tsc `node node_modules/typescript/bin/tsc --noEmit`, e2e `node node_modules/@playwright/test/cli.js test`(`PLAYWRIGHT_BASE_URL=http://127.0.0.1:3300`). 화면 캡처는 `.claude/launch.json`(node 직접 설정)으로 `preview_start` → 3300 점유 먼저 비울 것.
- **이관(무코드)**: 임베딩/LLM `base_url·model`을 사내 vLLM/`qwen3-30b-a3b`(MoE 30B/~3B활성, 회사 표준)로, `AGENT_FORGE_QDRANT_URL`만 교체.

## 알려진 한계 (배포 전)
SSO 미연동(헤더 스텁) · 프롬프트 인젝션은 약한 모델서 비결정적 우회(하드닝=베이스라인) · 업로드 TXT/MD/PDF/DOCX 지원(XLSX·원본파일 보관 미지원) · 객체저장소(MinIO)는 배선됨(AF-009, PR #30) — opt-in `AGENT_FORGE_OBJECT_STORE_BACKEND`(기본 none), 켜면 업로드 원본 보관 + 큐 잡이 스토어에서 fetch(풀 async 큐 분리는 보류) · run 조회 GET은 owner/admin 스코프(PR #29)지만 문서목록 GET 메타는 아직 무스코프(원문 없음).

(참고: 문서 ACL 편집/권한회수는 PR #21에서 구현됨 — `PATCH /documents/{id}/acl`, Qdrant payload 동기화 + 감사. 더는 한계 아님.)
