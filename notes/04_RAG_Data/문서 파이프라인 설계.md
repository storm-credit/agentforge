# 문서 파이프라인 설계

## 목표

MVP의 RAG 파이프라인은 사내망에서 문서를 안전하게 수집하고, 권한 메타데이터를 잃지 않은 채 검색 가능한 chunk로 만들며, 답변 시 citation으로 추적 가능한 근거를 제공하는 것을 목표로 한다. 구현은 화려한 자동화보다 재현성, 권한 안전성, 운영자가 설명할 수 있는 품질을 우선한다.

## 전체 흐름

```text
문서 수집
-> 원본 저장
-> 변경 감지/중복 제거
-> 파싱
-> 텍스트 정규화
-> 구조 추출
-> chunking
-> metadata/ACL 부여
-> embedding
-> lexical index
-> vector index
-> index snapshot 발행
-> ACL-aware retrieval
-> hybrid merge
-> rerank
-> citation 답변
-> 피드백/품질 로그
```

각 단계는 `job_id`, `document_id`, `source_id`, `content_hash`, `parser_version`, `chunker_version`, `embedding_model`, `index_snapshot_id`를 기록한다. 동일 원문과 동일 파이프라인 버전이면 같은 chunk ID가 나오도록 deterministic하게 설계한다.

## MVP 지원 형식

| 형식 | 처리 방식 | MVP 주의점 |
|---|---|---|
| PDF | 텍스트 레이어 우선, 필요 시 사내 OCR | 스캔 PDF는 OCR confidence를 metadata에 기록 |
| DOCX | 문단, heading, table 추출 | 변경 이력/주석은 기본 제외, 필요 시 별도 옵션 |
| XLSX | sheet/table 단위로 추출 | 계산식은 값과 formula를 분리 저장 |
| TXT | 인코딩 감지 후 line 기반 파싱 | 깨진 인코딩은 quarantine |
| Markdown | heading 계층 유지 | frontmatter를 metadata 후보로 사용 |

이미지, 동영상, CAD, 압축 파일은 MVP 직접 인덱싱 대상에서 제외한다. 단, 원본 보관소에는 저장하고 `unsupported_format` 상태로 남긴다.

## 1. 문서 수집

### 소스

- 사내 파일 서버 공유 폴더
- 그룹웨어 문서함 export
- 보안/인사/운영 부서가 관리하는 승인 문서 폴더
- 관리자가 수동 업로드한 문서

### 수집 계약

```yaml
source_id: "security_policy_docs"
source_type: "file_share"
root_path: "\\\\intranet\\policies\\security"
owner_department: "security"
sync_mode: "scheduled"
sync_interval_minutes: 60
allowed_extensions: [".pdf", ".docx", ".xlsx", ".txt", ".md"]
default_acl:
  department_scope: ["all"]
  role_scope: ["employee"]
  confidentiality_level: "internal"
```

수집기는 원본 파일을 수정하지 않는다. 원본은 WORM 성격의 object/file storage에 저장하고, 파싱 결과와 인덱스는 파생 산출물로 관리한다.

### 변경 감지와 중복 제거

- `content_hash`는 파일 바이트 기준 SHA-256으로 계산한다.
- 파일명만 바뀌고 내용이 같으면 새 문서가 아니라 alias로 기록한다.
- 내용이 바뀌면 새 `document_version`을 만들고 이전 버전은 검색 대상에서 제외하거나 유효 기간을 닫는다.
- 삭제 감지는 soft delete로 처리하고, audit 기간 동안 원본과 metadata를 보존한다.

## 2. 원본 저장

원본 저장소는 최소 다음 필드를 가진다.

```yaml
document_id: "sec-policy-2026"
document_version: "2026.05.01"
source_id: "security_policy_docs"
original_uri: "object://raw/security_policy/sec-policy-2026.pdf"
content_hash: "sha256-..."
mime_type: "application/pdf"
size_bytes: 830122
ingested_at: "2026-05-09T09:00:00+09:00"
ingested_by: "system"
retention_policy: "audit_3y"
```

원본 저장 실패 시 파싱을 진행하지 않는다. 파싱 결과만 있고 원본이 없는 상태는 재현성과 감사 추적이 깨지므로 허용하지 않는다.

## 3. 파싱

### 공통 원칙

- 문서 구조를 가능한 한 유지한다. heading, page, table, list, footnote, sheet 정보를 잃지 않는다.
- 파서 버전을 metadata에 기록한다.
- 파싱 confidence가 낮으면 인덱스에 바로 넣지 않고 quarantine queue로 보낸다.
- 문서 안의 숨김 텍스트, 주석, 변경 이력은 MVP 기본 검색에서 제외하고 별도 정책이 있을 때만 포함한다.

### 형식별 파싱

| 형식 | 구조 추출 | citation locator |
|---|---|---|
| PDF | page, text block, heading 추정, table 영역 | `page`, `block_id` |
| DOCX | heading level, paragraph, table, list | `heading_path`, `paragraph_index` |
| XLSX | workbook, sheet, table range, row/column | `sheet_name`, `cell_range` |
| TXT | line range, delimiter section | `line_start`, `line_end` |
| Markdown | heading path, code block, table | `heading_path`, `line_range` |

## 4. 텍스트 정규화

정규화는 검색 품질을 올리되 법적/감사상 의미를 바꾸지 않는 범위에서 수행한다.

- Unicode normalization은 NFC 기준으로 통일한다.
- 반복 공백, 페이지 머리말/꼬리말, 목차의 중복 링크를 제거한다.
- 표는 행/열 의미가 드러나도록 Markdown table 또는 key-value 문장으로 직렬화한다.
- 날짜, 문서 번호, 정책 조항 번호는 삭제하지 않는다.
- 개인정보는 인덱싱 정책에 따라 원문 유지, 마스킹, 제외 중 하나를 선택하고 metadata에 표시한다.

## 5. Chunking

### 기본 정책

| 항목 | 기준 |
|---|---|
| target chunk size | 650 tokens |
| min chunk size | 250 tokens |
| max chunk size | 900 tokens |
| overlap | 80-120 tokens |
| split 우선순위 | heading > paragraph > sentence > token |
| table chunk | 한 표를 가능하면 하나의 chunk로 유지 |
| code/formula chunk | 설명 문맥과 분리하지 않음 |

chunk는 의미 단위와 citation 단위를 동시에 만족해야 한다. 너무 작으면 rerank는 잘 되지만 답변 맥락이 부족하고, 너무 크면 citation이 흐려진다. MVP는 650 token 목표값으로 시작하고 golden set에서 조정한다.

### 구조 보존 규칙

- heading path는 모든 chunk에 붙인다.
- PDF page가 여러 페이지에 걸친 chunk는 `page_start`, `page_end`를 모두 기록한다.
- DOCX table은 caption, 직전 heading, 표 본문을 함께 묶는다.
- XLSX는 sheet별로 별도 문서처럼 다루되 workbook metadata를 유지한다.
- 조항 번호가 있는 정책 문서는 조항 경계를 우선 split한다.

### chunk ID

```text
chunk_id = {document_id}:{document_version}:{structure_locator}:{chunk_index}:{chunk_hash_prefix}
```

예:

```text
sec-policy-2026:2026.05.01:p12:c03:8f2a91
hr-handbook:2026.01.15:h2-remote-work:p03:12ab77
```

chunk ID는 citation, audit log, feedback, 재색인 비교의 기준이므로 안정적으로 생성한다.

## 6. Metadata/ACL 부여

### 필수 chunk metadata

```yaml
chunk_id: "sec-policy-2026:2026.05.01:p12:c03:8f2a91"
document_id: "sec-policy-2026"
document_version: "2026.05.01"
source_id: "security_policy_docs"
title: "외부 반출 보안 절차"
section: "3.2 로그 및 진단자료"
heading_path: ["외부 반출", "로그 및 진단자료"]
page_start: 12
page_end: 12
language: "ko"
content_hash: "sha256-..."
chunk_hash: "sha256-..."
created_at: "2026-05-01"
effective_from: "2026-05-01"
effective_to: null
indexed_at: "2026-05-09T09:10:00+09:00"
parser_version: "pdf-parser/1.0.0"
chunker_version: "semantic-chunker/1.0.0"
embedding_model: "internal-embedding-ko-v1"
acl:
  department_scope: ["all"]
  role_scope: ["employee"]
  project_scope: ["corp-policy"]
  confidentiality_level: "internal"
  allow_user_ids: []
  deny_user_ids: []
```

### ACL 상속 우선순위

```text
문서 내 명시 ACL
> 문서 관리 시스템 ACL
> 폴더/소스 기본 ACL
> source default_acl
> quarantine
```

ACL을 확인할 수 없는 문서는 검색 인덱스에 넣지 않는다. 기본 공개로 추정하는 방식은 금지한다.

## 7. Embedding

### 모델 기준

- 사내망에서 실행 가능한 embedding 모델을 사용한다.
- 한국어 업무 문서 검색 품질을 검증한 모델만 registry에 등록한다.
- 모델명, 버전, dimension, normalization 여부를 인덱스 metadata에 저장한다.
- embedding 모델이 바뀌면 새 vector index와 새 `index_snapshot_id`를 만든다.

### 입력 구성

embedding 입력은 chunk 본문만 넣지 않고 검색에 도움이 되는 구조 정보를 함께 넣는다.

```text
title: 외부 반출 보안 절차
section: 3.2 로그 및 진단자료
content: 로그 파일과 진단자료를 외부 협력사에 전달하려면...
```

단, ACL/기밀 등급 같은 보안 metadata는 embedding text에 넣지 않는다. 보안 정보가 의미 검색으로 누출될 수 있기 때문이다.

## 8. Indexing

MVP는 vector index와 lexical index를 같이 둔다.

- vector index: 의미 검색, 유사 질문 대응
- lexical index: 문서 번호, 조항 번호, 고유명사, 약어 검색
- metadata store: 원본 locator, ACL, hash, lifecycle 상태

index 발행은 snapshot 단위로 한다.

```yaml
index_snapshot_id: "idx_security_policy_2026_05_01"
source_ids: ["security_policy_docs"]
embedding_model: "internal-embedding-ko-v1"
chunk_count: 18240
created_at: "2026-05-09T09:20:00+09:00"
status: "active"
quality:
  acl_field_coverage: 1.0
  parse_success_rate: 0.98
  quarantine_count: 17
```

## 9. Retrieval/Rerank/Citation

검색 시에는 다음 순서를 지킨다.

```text
권한 컨텍스트 생성
-> 쿼리 정규화/확장
-> ACL payload filter 생성
-> vector search + lexical search
-> hybrid merge(RRF)
-> rerank
-> citation 가능한 chunk만 선택
-> LLM context 구성
```

rerank는 cross-encoder 또는 사내 LLM scoring 중 사내망에서 운영 가능한 방식을 쓴다. MVP는 latency를 위해 top 40-60 후보까지만 rerank한다.

citation 후보는 다음 필드가 있어야 한다.

```yaml
document_id: "sec-policy-2026"
title: "외부 반출 보안 절차"
section: "3.2 로그 및 진단자료"
page: 12
chunk_id: "sec-policy-2026:2026.05.01:p12:c03:8f2a91"
source_uri_ref: "object://raw/security_policy/sec-policy-2026.pdf"
```

사용자에게는 내부 storage URI를 노출하지 않고 제목, 섹션, 페이지, 문서 버전 정도만 보여준다.

## 10. 품질/운영 게이트

- parse success rate가 소스별 95% 미만이면 index snapshot을 자동 활성화하지 않는다.
- ACL 필수 필드 coverage가 100%가 아니면 검색 인덱스에 발행하지 않는다.
- chunk 중 `title`, `source_id`, `document_id`, `chunk_id`, `citation_locator`가 빠진 항목이 있으면 실패 처리한다.
- golden query set에서 top-k recall, citation locator 정확도, 권한 차단 케이스를 통과해야 한다.
- quarantine 문서는 운영자가 원인과 조치 상태를 볼 수 있어야 한다.
