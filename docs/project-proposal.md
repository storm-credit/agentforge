## 요약

Agent Forge는 사내망 환경에서 승인된 모델, 사내 문서, 권한정책, 실행 로그를 조합해 업무용 AI 에이전트를 만들고 운영하는 내부 Agent Builder 플랫폼이다.

1차 MVP는 "사내 문서 기반 RAG 에이전트 빌더"로 제한한다. ERP, 그룹웨어, 결재 같은 쓰기 자동화로 바로 확장하지 않고, 먼저 권한 기반 검색, 근거 citation, 에이전트 설정, audit log를 검증한다.

## 문제 정의

| 문제 | 현재 현상 | 영향 |
|---|---|---|
| 데이터 반출 우려 | 사내 문서와 업무 데이터를 외부 AI SaaS에 입력하기 어렵다 | AI 활용 범위가 제한됨 |
| 권한 통제 부재 | 사용자가 보면 안 되는 문서까지 검색될 위험이 있다 | 정보 노출과 보안 감사 리스크 |
| 챗봇 난립 | 부서별 임시 챗봇이 별도로 만들어질 수 있다 | 중복 개발, 품질 편차, 유지보수 비용 증가 |
| 근거/감사 부족 | 답변 출처와 실행 기록이 남지 않는다 | 업무 활용 신뢰도와 책임 추적성 저하 |
| 확장 구조 부족 | 단발성 챗봇은 DB, 그룹웨어, ERP 에이전트로 확장하기 어렵다 | PoC 이후 운영 전환 실패 가능성 |

## 제안 내용

Agent Forge를 단일 챗봇이 아니라 재사용 가능한 사내 Agent Builder 플랫폼으로 구축한다.

- Agent Builder: 에이전트 목적, 모델, 문서, 권한, 답변 정책 설정
- Knowledge/RAG: 문서 수집, 파싱, 청킹, 임베딩, 검색, citation
- Policy Layer: 사용자, 부서, 그룹, 문서등급 기반 접근 제어
- Runtime Orchestrator: 질문, 검색, 답변 생성, 로그 저장 흐름 관리
- Audit Log: 사용자 질문, 검색 문서, 모델, 정책 버전, 답변 상태 추적

## MVP 범위

### 포함

| 영역 | MVP 포함 내용 |
|---|---|
| 에이전트 설정 | 이름, 목적, 소유자, 문서 묶음, 모델, 답변 정책, 접근 범위 |
| 문서 수집 | 파일럿 문서 묶음 기준 PDF, DOCX, XLSX, TXT, Markdown 처리 |
| 검색 | 답변 생성 전 ACL 기반 검색 후보 제한 |
| 답변 | 문서명과 위치가 포함된 citation, 근거 부족 시 한계 안내 |
| 운영 | Agent Card, 버전, 게시 상태, 색인 상태, 기본 관리자 화면 |
| 감사 | 질문, 검색 문서, 답변 상태, 모델, 정책 버전, 오류 로그 |
| 평가 | 평가 질문, citation 포함률, 유용성, 권한 차단 테스트 |

### 제외

- ERP/그룹웨어 쓰기 작업
- 메일 발송과 결재 자동 처리
- 전사 모든 문서 저장소의 자동 연결
- 완전 자율 멀티 에이전트 실행
- 고급 모델 라우팅과 비용 최적화
- 문서 등급/민감정보 자동 판정의 완전 자동화

## 핵심 MVP 유스케이스

관리자가 승인된 사내 문서 묶음, 모델, 답변 정책, 접근 권한을 설정해 RAG 기반 Q&A 에이전트를 게시한다. 사용자는 업무 질문을 입력하고, 본인이 접근 가능한 문서 안에서만 근거가 포함된 답변을 받는다.

권장 파일럿 후보는 다음과 같다.

| 후보 | 적합 이유 |
|---|---|
| 보안/개인정보 규정 Q&A | 권한, 감사, 정책 근거 답변 검증에 적합 |
| 인사/총무 업무 매뉴얼 Q&A | 반복 문의 감소 효과가 명확함 |
| IT 운영 절차 Q&A | 운영 지식 검색에 유용하나 문서 최신성 관리가 중요 |

최종 파일럿은 1주차에 문서 확보 가능성과 담당 부서 참여 의지를 기준으로 선정한다.

## 성공 기준

| 구분 | 지표 | 목표 |
|---|---|---|
| 보안 | 권한 없는 문서 노출 | 파일럿 테스트 기준 0건 |
| 신뢰도 | citation 포함률 | 평가 질문 기준 95% 이상 |
| 유용성 | 정답/유용 답변 비율 | 평가 질문 기준 80% 이상 |
| 운영성 | 관리자가 신규 파일럿 에이전트 설정 가능 | 가이드 기반으로 가능 |
| 추적성 | 주요 실행 단계 로그 | 질문, 검색, 답변, 오류, 모델, 정책 버전 기록 |
| 파일럿 전환 | v0.2 백로그 도출 | 12주차 리뷰 전 필수 |

## 8~12주 추진 계획

| 기간 | 목표 | 주요 산출물 |
|---|---|---|
| 1~2주 | 범위, 유스케이스, 보안/데이터 기준 확정 | 제안서 v0.1, 유스케이스 정의, 파일럿 부서, 문서 후보, ACL 초안 |
| 3~4주 | 설계 확정과 개발 골격 구성 | Architecture v0.1, Agent Registry/API 초안, 문서 파이프라인 골격 |
| 5~6주 | 핵심 MVP 기능 연결 | ACL 검색, citation 답변, Agent Studio 기본 화면, audit log |
| 7~8주 | MVP 통합 검증 | E2E 데모, 평가 결과, 권한 차단 테스트, MVP 데모 리포트 |
| 9~10주 | 파일럿 안정화 | 문서 추가, 품질/성능 개선, 운영 가이드, 사용자 피드백 |
| 11~12주 | 운영 전환 판단 | 파일럿 결과 보고, 보안/운영 리뷰, v0.2 백로그, 확장 로드맵 |

## 주요 리스크

| 리스크 | 대응 |
|---|---|
| MVP 범위가 ERP/그룹웨어 자동화까지 확대됨 | 1차 범위를 문서 RAG 에이전트 빌더로 고정하고 자동화 요구는 v0.2 백로그로 분리 |
| 권한 모델 확정이 지연됨 | 실제 문서 색인 전 ACL Matrix를 먼저 확정 |
| 파일럿 문서 확보가 늦어짐 | 1주차에 문서 소유자를 지정하고 대체 샘플 문서 준비 |
| 답변 품질이 부족함 | 평가 질문 세트를 조기에 만들고 검색/citation 정책을 반복 개선 |
| audit log에 민감정보가 과다 저장됨 | 로그 항목, 보관 기간, 마스킹 기준을 보안 담당자와 사전 합의 |
