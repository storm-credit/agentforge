# 폐쇄망 배포 구상

## 1. 목표

Agent Forge는 인터넷이 차단된 고객 내부망에서 재현 가능하게 설치, 업그레이드, 운영되어야 한다. MVP 배포 단위는 FastAPI Backend, Next.js Frontend, PostgreSQL, Qdrant 또는 pgvector, MinIO, vLLM 기반 Model Serving, Observability stack이다.

핵심 원칙:

- 모든 산출물은 외부 staging 환경에서 build 후 checksum/SBOM/서명과 함께 반입한다.
- 폐쇄망 내부에서는 인터넷 다운로드 없이 설치된다.
- 모델, 패키지, 컨테이너 이미지, DB migration, vector index를 버전으로 추적한다.
- 장애 시 원본 문서와 DB만으로 vector index를 재생성할 수 있어야 한다.

## 2. 배포 토폴로지

### 단일 노드 개발/PoC

```text
one host
  docker compose
    web-nextjs
    api-fastapi
    worker-indexing
    postgres
    qdrant 또는 postgres+pgvector
    minio
    vllm
    prometheus
    grafana
```

용도:

- 내부 PoC
- QA 평가 환경
- 작은 문서 corpus 검증

제약:

- GPU memory와 vector DB resource contention 가능
- 운영 SLA를 보장하기 어렵다.

### 운영 권장 분리

```text
zone: app
  frontend-nextjs
  backend-fastapi
  indexing/eval workers

zone: data
  postgresql primary/replica
  qdrant cluster 또는 postgres pgvector
  minio

zone: model
  vllm chat serving
  embedding serving
  reranker serving

zone: ops
  prometheus
  grafana
  loki 또는 opensearch
  backup repository
```

네트워크:

- Frontend -> Backend: HTTPS 내부 LB
- Backend -> PostgreSQL/Qdrant/MinIO/Model Gateway: allowlist
- 사용자 브라우저는 DB/vector/model 서버에 직접 접근하지 않는다.
- 운영 콘솔은 admin subnet 또는 VPN allowlist를 요구한다.

## 3. 구성 요소

| 영역 | 기본 선택 | 대안 | 비고 |
|---|---|---|---|
| Backend | FastAPI + Uvicorn/Gunicorn | FastAPI 단독 Uvicorn | API/worker 분리 |
| Frontend | Next.js standalone output | static export 일부 | App Router SSR 필요 시 Node runtime |
| DB | PostgreSQL 16+ | 고객 표준 PostgreSQL | migration은 Alembic |
| Vector | Qdrant | pgvector | 대규모 corpus는 Qdrant 우선 |
| Object | MinIO | 사내 S3 호환 스토리지 | 원본/파싱 artifact |
| Queue | Redis/RQ 또는 Celery | PostgreSQL job table | MVP는 job table polling 가능 |
| Model | vLLM OpenAI-compatible API | Ollama, 사내 gateway | 운영은 vLLM 우선 |
| Embedding | TEI/vLLM embedding endpoint | sentence-transformers worker | 모델별 adapter |
| Reranker | TEI 또는 custom FastAPI | 없음 | MVP 초기에는 optional |
| Observability | OpenTelemetry + Prometheus + Grafana | 고객 표준 APM | trace id를 run_id와 연결 |

- compatible migration range

## 5. 반입 절차

1. 외부 build 환경에서 release bundle 생성
2. SAST/dependency scan/container scan 실행
3. SBOM과 checksum 생성
4. 승인자가 bundle manifest 검토
5. 이동식 매체 또는 고객 승인 전송 채널로 폐쇄망 반입
6. 폐쇄망 staging에서 checksum 검증
7. 사내 registry에 image load/push
8. 사내 package mirror 또는 local wheelhouse/pnpm store 등록
9. 모델 파일 hash 검증 및 모델 registry row 생성
10. staging smoke test 후 운영 반영

검증 명령 예시:

```bash
sha256sum -c checksums.sha256
cosign verify --key agentforge.pub registry.local/agentforge/backend-api:YYYYMMDD.N
docker load -i images/backend-api.tar
```

## 6. 모델 서빙 계획

### vLLM Chat Serving

권장:

- OpenAI-compatible endpoint 사용: `/v1/chat/completions`
- 모델별 GPU node pool 분리
- tensor parallel은 모델 크기와 GPU 수에 따라 설정
- max model len은 RAG context와 citation 정책에 맞춰 제한
- request timeout과 max concurrent requests를 backend에서 제어

운영 설정:

- `served_model_name`: Agent Studio에 표시되는 이름과 일치
- `gpu_memory_utilization`: 초기 0.85 이하
- `max_num_seqs`: latency와 throughput 측정 후 조정
- `trust_remote_code`: 기본 false. 필요한 모델만 보안 승인 후 true

### Embedding Serving

옵션:

- vLLM/TEI embedding endpoint
- 별도 FastAPI embedding worker
- CPU fallback은 소량 문서에만 허용

Embedding 모델 변경 시:

- `embedding_model_version`을 새로 발급
- Qdrant collection 또는 pgvector 컬럼/index를 새로 만든다.
- 기존 index를 덮어쓰지 말고 blue/green 전환한다.

### Reranker Serving

MVP 초기에는 optional이지만 API/DB는 `rerank_top_k`와 score를 저장한다.

- reranker 장애 시 policy에 따라 degraded mode 선택
- citation required agent는 reranker 없이도 min_score 기준을 보수적으로 적용

### Model Gateway

Backend가 모델 서버를 직접 호출하지 않고 내부 `ModelGatewayClient`를 통해 호출한다.

필수 기능:

- 모델별 endpoint routing
- timeout/retry/circuit breaker
- prompt/token usage 기록
- 정책상 금지 모델 차단
- 모델 상태 health cache

## 7. Qdrant vs pgvector 운영 기준

### Qdrant 선택 기준

- 문서 chunk가 수십만 이상으로 증가할 가능성이 높다.
- payload ACL filter 조합이 많다.
- vector snapshot/restore와 collection alias 전환이 필요하다.
- DB 부하와 vector 검색 부하를 분리하고 싶다.

운영:

- collection alias: `chunks_active`
- 신규 embedding 모델 적용 시 `chunks_YYYYMMDD_model` collection 생성 후 alias 전환
- snapshot을 MinIO/backup repository에 저장

### pgvector 선택 기준

- 운영 단순성이 가장 중요하다.
- corpus가 작고 PostgreSQL 운영 역량이 강하다.
- 별도 Qdrant 운영 승인이 어렵다.

운영:

- HNSW index build 시간이 길 수 있으므로 maintenance window 확보
- VACUUM/ANALYZE와 index bloat 모니터링 필요
- payload filter 성능을 실제 ACL 조건으로 부하 테스트한다.

## 8. Kubernetes/Compose 배포

### Docker Compose MVP

PoC와 개발 폐쇄망은 compose로 시작한다.

필수 volume:

- `postgres_data`
- `qdrant_storage`
- `minio_data`
- `prometheus_data`
- `grafana_data`
- `model_cache`

필수 secret:

- DB password
- MinIO access/secret key
- JWT/SSO secret
- model gateway token

### Kubernetes 운영

권장 리소스:

- API: Deployment + HPA
- Worker: Deployment. indexing/eval queue별 분리
- Frontend: Deployment
- PostgreSQL/MinIO/Qdrant: 고객 표준 managed 또는 StatefulSet
- vLLM: GPU nodeSelector, toleration, PVC model cache

배포 전략:

- API/Frontend: rolling update
- Worker: drain 후 update
- DB migration: pre-deploy job. backward compatible migration 우선
- Model/Vector: blue/green 또는 alias switch

## 11. 로그/트레이싱

- OpenTelemetry trace id를 API request, run, run_step, model call에 전파한다.
- application log는 JSON으로 출력한다.
- 민감정보는 log ingestion 전 masking한다.
- run input/answer 전문 저장은 정책 옵션으로 제어한다.
- audit event는 별도 retention과 접근 권한을 둔다.

로그 필드:

```json
{
  "timestamp": "2026-05-09T09:00:00+09:00",
  "level": "INFO",
  "service": "backend-api",
  "request_id": "req_...",
  "trace_id": "trace_...",
  "run_id": "run_...",
  "event": "retrieval.completed",
  "latency_ms": 123,
  "user_id_hash": "..."
}
```

## 12. 백업/복구

백업 대상:

- PostgreSQL logical/physical backup
- MinIO bucket: raw documents, parsed artifacts, reports
- Qdrant snapshots 또는 pgvector DB backup
- Grafana dashboards/config
- release bundle과 manifest
- model manifest와 checksum

RPO/RTO 초안:

| 대상 | RPO | RTO | 비고 |
|---|---|---|---|
| PostgreSQL | 15분~1시간 | 4시간 | 고객 인프라 기준 조정 |
| MinIO | 1시간 | 4시간 | 원본 문서 필수 |
| Qdrant/pgvector | 24시간 | 8시간 | 원본+chunk로 재생성 가능 |
| Model files | release 단위 | 2시간 | bundle 재반입 가능 |

복구 우선순위:

1. PostgreSQL 복구
2. MinIO 원본/파싱 artifact 복구
3. API/Frontend 복구
4. Vector DB snapshot 복구 또는 index rebuild
5. Model serving 복구
6. eval quick suite로 품질 확인

## 13. 인덱스 재생성 절차

트리거:

- embedding 모델 변경
- chunking 정책 변경
- vector DB 손상
- ACL payload schema 변경

절차:

1. 신규 collection/table 준비
2. `documents.status=indexed` 문서 대상 rebuild job 생성
3. MinIO 원본 또는 parsed artifact에서 chunk 재생성
4. embedding 생성 및 upsert
5. sample query/eval suite 검증
6. alias 또는 config를 신규 index로 전환
7. 이전 index read-only 보관 후 삭제

주의:

- 재생성 중 운영 검색은 기존 active index를 사용한다.
- ACL 변경은 vector payload와 DB ACL이 불일치하지 않도록 version marker를 둔다.
