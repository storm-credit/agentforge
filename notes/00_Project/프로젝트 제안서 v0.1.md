## 1. 한 줄 제안

Agent Forge는 사내망 환경에서 승인된 모델, 사내 문서, 권한정책, 실행 로그를 조합해 업무용 AI 에이전트를 만들고 운영하는 내부 Agent Builder 플랫폼이다.

## 2. 제안 배경

외부 SaaS형 AI 도구를 그대로 도입하기 어려운 조직에서는 다음 문제가 먼저 해결되어야 한다.

| 문제 | 현상 | 방치 시 영향 |
|---|---|---|
| 데이터 반출 우려 | 사내 문서와 업무 데이터를 외부 도구에 입력하기 어렵다 | AI 활용 범위가 공개 자료 수준에 머무름 |
| 권한 통제 부재 | 사용자가 보면 안 되는 문서까지 검색될 위험이 있다 | 정보 노출, 보안 감사 리스크 |
| 챗봇 난립 | 부서별로 임시 챗봇을 별도 구축한다 | 유지보수 중복, 품질 편차, 운영 비용 증가 |
| 근거와 감사 부족 | 답변 출처, 검색 문서, 실행 단계가 남지 않는다 | 업무 활용 신뢰도와 책임 추적성 저하 |
| 확장 구조 부재 | 문서 Q&A 이후 DB, 그룹웨어, ERP 연동으로 확장하기 어렵다 | PoC 이후 운영 전환 실패 가능성 |

Agent Forge는 개별 챗봇을 하나 더 만드는 프로젝트가 아니라, 반복적으로 사내 AI 에이전트를 만들 수 있는 표준 운영 기반을 만드는 프로젝트로 제안한다.

## 3. 프로젝트 목표

### 3.1 비즈니스 목표

- 사내 문서를 기반으로 한 업무 질의응답을 안전하게 제공한다.
- 부서별 AI 에이전트 구축 방식을 표준화해 중복 개발을 줄인다.
- 답변 근거, 권한 검증, 실행 로그를 남겨 보안/감사 대응력을 확보한다.
- 향후 DB 조회, 그룹웨어, ERP 연동 에이전트로 확장 가능한 기반을 만든다.

### 3.2 MVP 목표

1차 MVP는 "사내 문서 기반 RAG 에이전트 빌더"로 제한한다. 관리자는 승인된 문서 묶음, 모델, 검색 정책, 답변 정책, 접근 권한을 설정해 파일럿용 문서 Q&A 에이전트를 생성하고, 사용자는 권한이 허용된 문서 범위 안에서 근거가 포함된 답변을 받는다.

## 4. 제안 범위

### 4.1 포함 범위

| 영역 | MVP 포함 내용 |
|---|---|
| Agent Builder | 에이전트 이름, 목적, 소유자, 문서 묶음, 모델, 답변 정책 설정 |
| Knowledge/RAG | PDF, DOCX, XLSX, TXT, Markdown 기본 수집/파싱/청킹/임베딩 |
| 권한 통제 | 사용자/부서/문서등급 기반 검색 필터와 접근 정책 적용 |
| 답변 품질 | 근거 citation, 모르면 모른다고 답변, 원문 링크/문서명 표시 |
| 운영 관리 | 에이전트 카드, 버전, 게시 상태, 색인 상태 확인 |
| 감사 로그 | 질문, 검색 문서, 답변, 모델, 정책 버전, 오류 로그 저장 |
| 평가 | 샘플 질문 세트, 답변 유용성, citation 포함률, 권한 차단 테스트 |

### 4.2 제외 범위

- ERP, 그룹웨어, 결재 시스템의 쓰기 작업
- 메일 자동 발송, 결재 자동 처리, 예약 실행
- 전사 모든 문서의 일괄 연결
- 완전 자율 멀티 에이전트 실행
- 고급 모델 라우팅, 비용 최적화, 다국어 품질 고도화
- 민감정보 자동 마스킹의 완전 자동화

## 5. 핵심 사용자

| 사용자 | 주요 니즈 | MVP에서 제공할 가치 |
|---|---|---|
| 업무 부서 담당자 | 반복 문의를 줄이고 최신 문서를 쉽게 찾고 싶다 | 문서 기반 Q&A 에이전트로 업무 문의 대응 |
| 에이전트 관리자 | 문서와 정책을 조합해 빠르게 에이전트를 만들고 싶다 | Agent Studio에서 설정, 게시, 버전 관리 |
| 보안/감사 담당자 | 권한 없는 문서 노출과 근거 없는 답변을 막고 싶다 | ACL 검색 필터, citation, audit log |
| AI/IT 운영자 | 모델, 색인, 실행 장애를 통제하고 싶다 | 모델 게이트웨이, 색인 상태, 실행 로그 |
| 경영/부서장 | PoC가 운영 가능한 구조인지 확인하고 싶다 | 파일럿 지표와 8~12주 로드맵 |

## 6. 대표 MVP 시나리오

### 시나리오: 사내 규정/업무 매뉴얼 Q&A 에이전트

1. 관리자가 파일럿 부서의 공개 가능 문서를 등록한다.
2. 시스템이 문서를 파싱하고 문단 단위로 색인한다.
3. 관리자가 에이전트 목적, 접근 부서, 답변 정책, 모델을 설정한다.
4. 사용자가 "출장비 정산 시 증빙 기준은 무엇인가요?"처럼 질문한다.
5. 런타임은 사용자 권한을 확인하고 허용된 문서에서만 검색한다.
6. 답변은 문서명, 섹션, 페이지 또는 원문 위치를 근거로 함께 제공된다.
7. 검색 문서, 답변, 정책 버전, 사용자 피드백이 audit log에 남는다.

## 7. 성공 기준

| 구분 | 지표 | 목표 |
|---|---|---|
| 보안 | 권한 없는 문서 노출 | 파일럿 테스트 기준 0건 |
| 답변 신뢰도 | citation 포함률 | 평가 질문 기준 95% 이상 |
| 업무 유용성 | 정답/유용 답변 비율 | 평가 질문 기준 80% 이상 |
| 운영성 | 신규 에이전트 설정 가능 여부 | 관리자 1명이 가이드에 따라 설정 가능 |
| 추적성 | 주요 실행 단계 로그 | 질문, 검색, 답변, 오류, 정책 버전 기록 |
| 파일럿 전환 | 시범 부서 사용성 검토 | 주요 개선 항목과 v0.2 백로그 도출 |

## 8. 8~12주 추진안

| 기간 | 목표 | 주요 산출물 |
|---|---|---|
| 1~2주 | 범위/요구사항 확정 | 제안서 v0.1, 유스케이스 정의, 파일럿 부서/문서 후보, ACL 초안 |
| 3~4주 | 설계와 골격 구현 | 아키텍처 v0.1, Agent Registry/API 초안, 문서 파이프라인 골격 |
| 5~6주 | 핵심 기능 연결 | ACL 기반 검색, citation 답변, Agent Studio 기본 화면, audit log |
| 7~8주 | MVP 통합 검증 | E2E 데모, 평가 질문 세트, 보안 차단 테스트, MVP 데모 리포트 |
| 9~10주 | 파일럿 안정화 | 문서 추가, 성능/품질 개선, 운영 가이드, 사용자 피드백 |
| 11~12주 | 운영 전환 판단 | 파일럿 결과 보고, v0.2 범위, 확장 우선순위, 의사결정 자료 |

상세 일정은 [[notes/01_PM/WBS|WBS]]를 기준으로 관리한다.

## 9. 주요 리스크와 대응 방향

| 리스크 | 대응 방향 |
|---|---|
| MVP 범위가 ERP/그룹웨어까지 확대됨 | 1차 목표를 문서 RAG 에이전트 빌더로 고정하고, 쓰기 작업은 v0.2 이후로 분리 |
| 권한 모델 확정 지연 | 문서 수집 전 ACL Matrix와 테스트 계정을 먼저 확정 |
| 샘플 문서 확보 지연 | 파일럿 부서별 공개 가능 문서 후보를 1주차에 선정 |
| 답변 품질 부족 | 평가 질문 세트와 citation 기준을 먼저 만들고 반복 튜닝 |
| 감사 로그에 민감정보가 과다 저장됨 | 로그 저장 항목과 마스킹 기준을 보안 담당자와 선합의 |

상세 리스크는 [[notes/01_PM/리스크 이슈 로그|리스크 이슈 로그]]에서 관리한다.

## 10. 의사결정 요청 사항

- 1차 MVP 범위를 "사내 문서 기반 RAG 에이전트 빌더"로 승인할 것인가?
- 8주 MVP 데모와 12주 파일럿 결과 보고 일정을 기준으로 추진할 것인가?
- 파일럿 부서 1곳과 문서 소유자 1명을 지정할 것인가?
- 보안/인프라/AI 담당자가 1~2주차 설계 리뷰에 참여할 수 있는가?
- MVP 평가 기준을 권한 안전성, citation, 유용성, 운영성 중심으로 둘 것인가?
